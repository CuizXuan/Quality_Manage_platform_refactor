Quality Manage Platform 模块详细设计
==============================

1. 终端调试台

--------

### 1.1 功能说明

* 手工编辑 HTTP 请求（Method、URL、Query、Header、Cookie、Auth、Body）
* **自动识别粘贴的 fetch/cURL/抓包工具接口信息并自动填充**
* 支持 JSON、Form、Multipart、Raw 格式
* 响应体支持：
  * 状态码、耗时、响应头
  * JSON 格式化、表格视图、原始文本
  * 搜索、复制、历史对比
* **接口健壮性分析**：
  * 字段完整性、类型正确性、异常值检测
  * AI 自动生成断言建议
* 保存调试历史、收藏请求
* 保存为基础用例或追加为场景步骤

### 1.2 数据结构

    DebugRequest {  id: string  method: string  url: string  query_params: Record<string, string>  headers: Record<string, string>  cookies: Record<string, string>  auth_config: AuthConfig  body_type: 'json' | 'form' | 'multipart' | 'raw'  body: string  environment_id: string  created_by: string  created_at: datetime}DebugResult {  id: string  debug_request_id: string  status_code: number  response_headers: Record<string,string>  response_body: string  duration_ms: number  error_message?: string  created_at: datetime  robustness_score: number  ai_assertions?: Assertion[]}

### 1.3 接口设计

* `POST /debug/request` 创建调试请求
* `POST /debug/send` 发送请求并返回结果
* `GET /debug/result/{id}` 查询调试结果
* `POST /debug/save-to-testcase` 保存为基础用例
* `POST /debug/save-to-scenario` 追加为场景步骤
* `GET /debug/history` 查询历史请求

### 1.4 使用场景

* 快速调试接口，生成可复用测试资产
* 自动发现接口异常，生成健壮性评分
* 粘贴浏览器抓包 curl/fetch 自动填充请求

### 1.5 看板组件

* 最近调试成功率
* 高频失败接口
* 收藏请求列表

2. 用例中心与变体管理

------------

### 2.1 功能说明

* 基础用例增删改查、文件夹/标签管理、版本管理
* 请求配置、断言、变量、前后置脚本管理
* 从终端调试保存基础用例
* 派生变体（正常、边界、空值、缺失字段、类型错误、枚举非法、超长字段、鉴权失败、权限不足、返回结构校验、返回业务值校验、性能阈值）
* 批量执行与覆盖率统计

### 2.2 数据结构

    TestCase {  id: string  folder_id: string  name: string  description: string  method: string  url: string  query_params: Record<string,string>  headers: Record<string,string>  cookies: Record<string,string>  auth_config: AuthConfig  body_type: string  body: string  expected_status: number  source_debug_id?: string  created_by: string  created_at: datetime  updated_at: datetime}CaseVariant {  id: string  case_id: string  name: string  variant_type: string  override_params: Record<string,string>  override_headers: Record<string,string>  override_body?: string  expected_status?: number  expected_schema?: any  assertions?: Assertion[]  created_by: string  created_at: datetime}

### 2.3 接口设计

* `POST /testcase` 创建用例
* `GET /testcase/{id}` 查询用例详情
* `PUT /testcase/{id}` 更新用例
* `DELETE /testcase/{id}` 删除用例
* `POST /testcase/{id}/variant` 创建变体
* `GET /testcase/{id}/variant` 查询变体列表

### 2.4 使用场景

* QA 构建接口测试覆盖
* 自动生成异常/边界场景
* 回归测试保证变体覆盖率

### 2.5 看板组件

* 用例覆盖率统计
* 变体执行通过率
* 异常变体失败率趋势

3. 场景编排与执行历史

------------

### 3.1 功能说明

* 场景增删改查
* 步骤增删改查、排序、启停
* 步骤失败策略（stop/continue/retry/skip）、超时、重试次数
* 步骤间变量提取/注入（JSONPath/Header/Cookie/状态码/自定义脚本）
* 执行记录保存快照（请求/响应/断言/变量/版本/日志）

### 3.2 数据结构

    Scenario {  id: string  name: string  description: string  status: string  version: number  created_by: string  created_at: datetime}ScenarioStep {  id: string  scenario_id: string  case_id: string  variant_id?: string  name: string  sort_order: number  enabled: boolean  retry_count: number  timeout_ms: number  failure_strategy: 'stop'|'continue'|'retry'|'skip'  extract_rules: Rule[]  inject_rules: Rule[]}ExecutionRun {  id: string  run_type: 'case'|'scenario'  target_id: string  environment_id: string  status: string  started_at: datetime  finished_at: datetime  duration_ms: number  summary: string}

### 3.3 接口设计

* `POST /scenario` 创建场景
* `GET /scenario/{id}` 查询场景
* `PUT /scenario/{id}` 更新场景
* `DELETE /scenario/{id}` 删除场景
* `POST /scenario/{id}/execute` 执行场景
* `GET /execution/{id}` 查询执行历史

### 3.4 使用场景

* 端到端业务流程测试
* 回归测试
* 步骤失败定位

### 3.5 看板组件

* 场景执行通过率趋势
* 步骤失败热点分析
* 执行耗时分布图

4. 报告中心与缺陷中心

------------

### 4.1 功能说明

* 报告生成、列表、详情
* 数据统计：通过率、失败分布、耗时、变体/场景覆盖、缺陷关联
* 缺陷增删改查、状态流转、优先级、复测、修复版本
* 外部系统同步
* AI 自动归因失败、生成修复建议

### 4.2 数据结构

    Report {  id: string  execution_id: string  summary: string  pass_rate: number  fail_distribution: Record<string,number>  duration_stats: {avg:number,max:number}  variant_coverage: number  scenario_coverage: number  defect_ids: string[]  created_at: datetime}Defect {  id: string  name: string  status: string  severity: string  priority: string  linked_case_ids: string[]  linked_scenario_ids: string[]  created_by: string  created_at: datetime  updated_at: datetime  fix_version?: string  retest_result?: string}

### 4.3 接口设计

* `GET /report` 报告列表
* `GET /report/{id}` 报告详情
* `POST /defect` 创建缺陷
* `PUT /defect/{id}` 更新缺陷
* `GET /defect/{id}` 查询缺陷

### 4.4 看板组件

* 缺陷分布热力图
* 高危接口缺陷趋势
* AI 待处理缺陷列表

5. 质量门禁与 AI 中枢

--------------

### 5.1 功能说明

* 门禁规则管理（通过率、失败率、高危缺陷、覆盖率、性能）
* 阻断/警告/放行/人工豁免
* AI 辅助：
  * 自动生成用例、变体、断言
  * 执行失败归因
  * 报告总结和风险预测

### 5.2 数据结构

    QualityGate {  id: string  rule_type: string  threshold: number  status: 'pass'|'warn'|'block'  evaluation_time: datetime}AIAnalysis {  id: string  target_type: 'case'|'variant'|'scenario'|'execution'  target_id: string  analysis_type: string  payload: any  suggestions: AISuggestion[]}

### 5.3 接口设计

* `POST /quality-gate/evaluate` 计算门禁
* `GET /ai/suggestions` 获取 AI 建议
* `POST /ai/accept` 采纳 AI 建议

### 5.4 看板组件

* 门禁通过率趋势
* 风险接口提醒
* AI 分析潜在失败点
