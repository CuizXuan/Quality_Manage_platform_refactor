## 3. 页面设计原则

- 登录页保持现状
- 业务页使用 App Shell
- 页面是工作台，不是营销页
- 高密度但要有清晰层级
- 不使用大面积装饰图
- 不使用卡片套卡片
- 支持深色和浅色主题
- 系统基调为“运维控制台”：左侧导航、顶部上下文、右侧工作区
- 主色保持冷静克制，强调色只用于主按钮、选中态和关键状态
- 表格、抽屉、权限矩阵是后台管理页面的核心承载形式

### App Shell

- 左侧菜单承载全局导航，顶部只放当前页面标题和用户操作。
- 顶部主题切换、个人中心、退出登录使用图标按钮，并提供 `title` 提示。
- 页面标题使用中文，英文只作为辅助 kicker 或技术标识。
- 工作区不放营销式 Hero，首页工作台也应服务于数据扫描和任务入口。

### 多级菜单

- 菜单最多默认展示两级：一级为模块分组，二级为具体页面。
- 超过两级时不继续缩进侧栏，而是在目标模块页内使用 Tabs、树表或左侧二级面板承载。
- 一级菜单不直接绑定零散按钮权限，只表达信息架构。
- 菜单编码和权限编码保留英文，菜单名称、状态、操作展示使用中文。
- 侧栏菜单从树形数据渲染，后续接入后端菜单时保持 `children` 结构。
- 菜单管理页必须使用树表展示层级，支持展开、收起、新增子级和选择上级菜单。
- 编辑菜单时不能把当前菜单或当前菜单的后代作为自己的上级菜单。

### 长文本

- 表格、列表、导航、标签中的长文本必须单行截断，使用省略号展示。
- 被截断内容必须保留完整 `title`，方便悬停查看。
- 详情页、抽屉表单、代码编辑器允许多行展示，但必须限制最大高度并可滚动。
- 权限摘要、角色列表、描述、邮箱、路径、组件标识均按长文本处理。

### 权限展示

- 角色基础信息只维护角色名称、编码、描述、状态。
- 按钮权限使用独立“授权”动作打开权限矩阵，不混入基础信息表单。
- 权限矩阵行是菜单或资源，列是查看、新增、编辑、删除。
- 权限矩阵必须分页展示，避免菜单数量增加后形成超长滚动区域。
- 授权抽屉需要显示已选权限数、菜单总数和当前页码。
- 不存在的动作显示 `-`，不要用禁用复选框制造误解。
- 后端仍保存 `resource:action` 编码，前端只展示中文资源和中文动作。

## 5. Store 规范

每个领域一个 Store：

- `useAuthStore`
- `useSystemStore`
- `useTerminalStore`
- `useCaseStore`
- `useScenarioStore`
- `useEnvironmentStore`
- `useExecutionStore`
- `useAiStore`

Store 只负责状态和调用 API，不写复杂 UI 逻辑。

## 6. API 规范

每个领域一个 API 文件：

- `auth.js`
- `system.js`
- `terminal.js`
- `case.js`
- `scenario.js`
- `environment.js`
- `execution.js`
- `ai.js`

API 文件只封装请求，不处理页面状态。

## 7. 样式规范

- 统一使用 CSS 变量
- 字体不随视口缩放
- 中文使用 `PingFang SC` / `Microsoft YaHei`
- 英文数字使用 `Inter`
- 代码使用 `JetBrains Mono`
- 表格、按钮、抽屉、弹窗统一半径和间距
- 状态色统一：成功、失败、警告、进行中、禁用
- 图标按钮尺寸固定，默认 36px，不因文案变化造成布局抖动
- 表格单元格必须有最大宽度和省略号策略
- 深浅主题共用布局和层级，只替换 CSS 变量，不单独写两套结构

## 8. 交互规范

- 所有提交动作必须有 loading
- 所有删除动作必须二次确认
- 所有空状态必须给下一步动作
- 所有执行动作必须能查看日志
- 所有 AI 建议必须先预览再采纳
