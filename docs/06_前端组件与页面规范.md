质量管理平台前端样式与布局规范
===============

1. 总体原则

-------

* **UI 组件优先**：布局、表单、表格、弹窗等一律使用 Element Plus 组件，避免用原生 div 模拟。

* **官方能力优先检查**：开发任何页面或组件前，必须先检查 Vue 官方能力是否已覆盖，再检查 Element Plus 官方组件是否已覆盖；只有两者都无法满足业务语义时，才允许自定义封装。

* **CSS 变量驱动**：所有颜色、间距、圆角、阴影等视觉属性必须通过 CSS 变量定义，禁止硬编码色值。

* **scoped 隔离**：每个 Vue 组件的样式使用 `<style scoped>`，避免样式泄露；全局样式只存放主题变量、布局结构和 Element Plus 覆盖。

* **深浅色原生支持**：通过一套 CSS 变量 + 根类切换实现，不引入额外 UI 库。

* * *

2. 组件选型流程

------------

### 2.1 开发前检查顺序

开发任何新功能、页面或组件前，必须按以下顺序确认：

1. **先查 Vue 官方能力**
   * 是否可以通过 Vue 内置能力解决：`v-model`、`defineProps`、`defineEmits`、`computed`、`watch`、`slot`、`Teleport`、`Transition`、`KeepAlive`、动态组件等。
   * 如果只是状态同步、条件渲染、插槽组合、页面缓存、弹层挂载等问题，优先使用 Vue 官方能力，不额外封装奇怪组件。

2. **再查 Element Plus 官方组件**
   * 表单：`el-form`、`el-form-item`、`el-input`、`el-input-number`、`el-select`、`el-option`、`el-radio-group`、`el-radio-button`、`el-checkbox-group`、`el-switch`、`el-date-picker`。
   * 数据展示：`el-table`、`el-table-column`、`el-tag`、`el-descriptions`、`el-empty`、`el-pagination`、`el-tree`。
   * 布局与容器：`el-container`、`el-aside`、`el-header`、`el-main`、`el-row`、`el-col`、`el-tabs`、`el-card`、`el-scrollbar`。
   * 反馈与弹层：`el-dialog`、`el-drawer`、`el-popover`、`el-tooltip`、`ElMessage`、`ElMessageBox`、`el-alert`、`el-progress`、`el-skeleton`。
   * 导航：`el-menu`、`el-sub-menu`、`el-menu-item`、`el-breadcrumb`、`el-dropdown`。

3. **最后才允许自定义封装**
   * 只有当 Vue 官方能力和 Element Plus 官方组件都不能表达业务语义时，才新增自定义组件。
   * 自定义组件必须封装业务组合，不允许重新发明基础控件。例如不能自绘 select、checkbox、radio、table、drawer、dialog、tabs、menu。
   * 自定义前必须在代码注释或开发说明中写清楚：为什么 Vue / Element Plus 不能满足、封装边界是什么、是否保持 Element Plus 的交互和可访问性。

### 2.2 禁止自绘的基础控件

以下控件必须使用 Element Plus 官方组件：

| 场景 | 必须使用 |
|------|----------|
| 下拉选择 | `el-select` + `el-option` |
| 单选 | `el-radio-group` / `el-radio-button` |
| 多选 | `el-checkbox-group` / `el-checkbox` |
| 表单 | `el-form` + `el-form-item` |
| 输入 | `el-input` / `el-input-number` |
| 表格 | `el-table` + `el-table-column` |
| 抽屉 | `el-drawer` |
| 弹窗 | `el-dialog` |
| 菜单 | `el-menu` / `el-sub-menu` / `el-menu-item` |
| 标签页 | `el-tabs` / `el-tab-pane` |
| 面包屑 | `el-breadcrumb` / `el-breadcrumb-item` |
| 空状态 | `el-empty` |
| 分页 | `el-pagination` |

### 2.3 后续模块开发模式（必须参照）

后续所有模块开发，默认参照当前“终端调试台 / 系统管理 / 工作台”这套模式，不允许每个页面重新发明一套交互方式。

核心要求：

1. **基础控件始终使用官方组件**
   * 输入、下拉、单选、多选、表格、抽屉、弹窗、标签页、菜单、分页、面包屑、消息提示，全部优先使用 Element Plus 官方组件。
   * 不允许因为“看起来更自由”就手写一个假的选择器、假的抽屉、假的 toast。

2. **能复用的逻辑抽为业务组合组件**
   * 允许抽取“业务组合组件”，但前提是其内部仍然由 Vue + Element Plus 官方组件组成。
   * 例如：请求参数编辑器、响应查看器、鉴权编辑器、历史记录项、分析面板。
   * 组合组件的目标是复用业务结构，不是替代官方基础控件。

3. **提示反馈统一**
   * 成功、失败、警告、确认类反馈统一使用 Element Plus 的 `ElMessage` / `ElMessageBox`。
   * 所有关键操作都必须有明确反馈：
     * 收藏成功 / 取消收藏成功
     * 保存成功 / 删除成功 / 更新成功
     * 发送成功 / 导入成功 / 解析成功
     * 刷新成功 / 加载成功
   * 后续建议将散落的 `ElMessage` 调用进一步收口到统一反馈工具，例如 `utils/feedback.ts`，但底层仍然调用官方能力。

4. **页面结构保持一致性**
   * 列表页：`el-table + el-pagination + toolbar`
   * 编辑页：`el-form`
   * 详情/配置页：`el-tabs` 或分区卡片
   * 弹出编辑：`el-drawer` 或 `el-dialog`
   * 页面内状态切换、筛选、批量操作的交互方式应与现有模块保持一致，不允许同类功能多个页面多种写法。

5. **封装边界清晰**
   * 公用组件只封装重复出现两次及以上的业务结构。
   * 纯页面私有逻辑不强行抽离。
   * 一个组件如果只是为了“少写几行模板”而把官方组件包得过深，属于过度封装，不建议这样做。

### 2.4 推荐的可复用组件方向

以下类型优先考虑沉淀为公用业务组件：

| 场景 | 推荐封装形式 |
|------|--------------|
| 请求参数编辑 | `KeyValueEditor` |
| 鉴权配置 | `AuthEditor` |
| 请求体编辑 | `BodyEditor` |
| 响应展示 | `ResponseBodyViewer` / `ResponseHeadersViewer` / `ResponseInfoViewer` |
| 保存为用例 | `SaveToCaseDialog` |
| 通用 CRUD 抽屉表单 | 在 `SystemCrudPage` 级别复用，但内部仍是 `el-table + el-drawer + el-form` |
| 列表过滤条 | 统一 toolbar 模式 |

### 2.5 不推荐的封装方式

以下封装方式应避免：

* 把 `el-select` 再包装成一个“自定义选择器”，但没有新增业务价值。
* 手写 toast/提示条代替 `ElMessage`。
* 手写 modal/drawer 框架代替 `el-dialog` / `el-drawer`。
* 为了追求“统一风格”而把 `el-table` 重写成 div 表格。
* 为了减少模板代码而把大量官方组件深包裹，导致后续维护困难。

### 5.2 左侧菜单（`MenuSidebar.vue`）

* 使用 **Element Plus 的 `el-menu`** 组件，属性：`collapse`（支持折叠）、`background-color`、`text-color`、`active-text-color`。

* 菜单数据来自后端或前端静态配置，结构为树形（`children`）。

* **多级菜单规则**：
  
  * 一级菜单分组（无 `children` 的直接作为可点击页面）。
  
  * 超过两级时，不再在侧栏嵌套缩进，而是在目标页面内使用二级 Tabs 或左侧子面板（例如用例中心采用 `el-tabs` 区分“用例列表”和“用例详情”）。

* 菜单项必须包含 `icon`（使用 Element Plus 图标库）、`index`（路由路径）、`title`（中文名称）。

vue

<template>
  <el-menu
    :default-active="$route.path"
    :collapse="isCollapse"
    :unique-opened="true"
    background-color="var(--bg-sidebar)"
    text-color="var(--text-primary)"
    active-text-color="var(--el-color-primary)"
  >
    <el-sub-menu v-for="item in menuList" :key="item.path" :index="item.path">
      <template #title>
        <el-icon><component :is="item.icon" /></el-icon>
        <span>{{ item.title }}</span>
      </template>
      <el-menu-item v-for="child in item.children" :index="child.path">
        <el-icon><component :is="child.icon" /></el-icon>
        <span>{{ child.title }}</span>
      </el-menu-item>
    </el-sub-menu>
  </el-menu>
</template>

* **高亮与选中态**：使用 `el-menu` 自带的 `active-text-color` 为 `var(--el-color-primary)`，背景色不额外改变。

### 6.1 页面布局（以终端调试台为例）

每个页面内部使用 **Element Plus 的 `el-row` / `el-col`** 或 `flex` 实现分栏，不用手写复杂 CSS。

<template>

      </el-col>
      <el-col :span="16">

        <ResponsePanel />
        <AiAnalysisPanel />
      </el-col>
    </el-row>
  </div>

<style scoped>
.debugger-page {
  height: 100%;
  background-color: var(--bg-page);
}
/* 子组件内部样式也使用 scoped，且只使用 CSS 变量 */
</style>

### 6.2 组件内部样式写法

* 类名使用 **BEM 或短横线命名**（如 `.request-panel__header`）。

* 禁止使用 `!important`。

* 所有自定义样式必须基于 CSS 变量，例如：

css

.request-panel {
  border: 1px solid var(--border-color);
  border-radius: var(--border-radius-base);
  background-color: var(--bg-container);
  padding: var(--spacing-md);
  margin-bottom: var(--spacing-md);
}

* 对于表格、表单等复杂组件，优先使用 Element Plus 提供的属性（如 `stripe`、`border`），不自己造轮子。

### 6.3 长文本、截断与省略

全局类名：

css

7. 深浅色主题切换实现

------------

### 7.1 切换逻辑（`composables/useTheme.ts`）

typescript

import { ref, watch } from 'vue';
export const useTheme = () => {
  const isDark = ref(localStorage.getItem('theme') === 'dark');
  const setTheme = (dark: boolean) => {
    const html = document.documentElement;
    if (dark) {
      html.classList.add('dark');
    } else {
      html.classList.remove('dark');
    }
    localStorage.setItem('theme', dark ? 'dark' : 'light');
    isDark.value = dark;
  };
  // 监听系统配色（可选）
  const matchMedia = window.matchMedia('(prefers-color-scheme: dark)');
  matchMedia.addEventListener('change', (e) => {
    if (!localStorage.getItem('theme')) {
      setTheme(e.matches);
    }
  });
  return { isDark, setTheme, toggle: () => setTheme(!isDark.value) };
};

### 7.2 在顶部栏添加切换按钮

vue

<el-button text :icon="isDark ? Sunny : Moon" @click="toggleTheme" />

### 7.3 注意 Element Plus 暗色适配

Element Plus 内置暗色主题变量，但我们已通过 CSS 变量覆盖保持一致。确保在 `main.ts` 中引入：

js

import 'element-plus/theme-chalk/dark/css-vars.css';

这样 Element Plus 内部组件也会自动跟随 `.dark` 类变化。

### 8.2 菜单权限控制

* 菜单渲染时根据后端返回的权限列表过滤（在 store 中维护 `accessibleMenus`）。

* 无权限的菜单项不显示，子菜单全部无权限时父级也不显示。

### 8.3 折叠状态

* 左侧菜单支持折叠（宽度从 `240px` 变为 `64px`），图标 + `title` 隐藏。

* 折叠状态保存在 `localStorage` 或 pinia 中，保持用户偏好。

### 8.4 当前路由高亮

* `el-menu` 绑定 `:default-active="$route.path"`，当路由变化时自动高亮匹配项。

* 对于动态路由（如 `/case/detail/:id`），需设置 `index` 为父级路径或使用 `resolve` 方法。

10. 禁止事项（AI 或开发者必须遵守）

--------------------

* ❌ 禁止在任何 `.vue` 文件的 `<style scoped>` 中定义全局 CSS 变量。

* ❌ 禁止在组件内硬编码颜色（如 `color: red`、`background: #fff`）。

* ❌ 禁止使用 `!important` 覆盖 Element Plus 默认样式（应通过 CSS 变量修改）。

* ❌ 禁止使用非 Element Plus 的栅格系统（如 Bootstrap、自写 flex 网格）。

* ❌ 禁止在菜单栏中使用 `el-menu` 以外的自定义列表模拟菜单。

* ❌ 禁止在深色主题下遗漏任何文本或背景变量，导致不可读。

* ❌ 禁止在没有先检查 Vue 官方能力和 Element Plus 官方组件的情况下自定义基础控件。

* ❌ 禁止自绘 select、checkbox、radio、table、drawer、dialog、tabs、menu、breadcrumb 等 Element Plus 已提供的组件。

* ❌ 禁止不同模块各自定义不同风格的提示反馈方式；成功/失败反馈必须统一走 `ElMessage` / `ElMessageBox`。

* ❌ 禁止为了抽象而抽象；如果封装后的组件没有明显复用价值或业务语义提升，就不应新增公用组件。

11. AI 生成代码时的检查清单

-----------------

当你（AI）生成任何新页面或组件时，必须确认：

1. ✅ 已先检查 Vue 官方能力是否可满足当前需求。
