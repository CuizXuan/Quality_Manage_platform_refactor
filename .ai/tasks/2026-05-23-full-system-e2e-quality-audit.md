# 全系统功能串联测试与质量管理完整性审计

## 任务目标

请对当前质量管理平台做一次完整的功能串联测试和产品完整性审计，重点验证“用例管理 -> 场景编排 -> 场景执行 -> 执行历史/报告 -> 缺陷中心 -> 质量门禁 -> AI 中枢辅助能力”的主链路是否能闭环运行。

本任务以审计、验证、分析为主，不要求直接修复代码。除非遇到极小范围、明显影响审计继续进行的配置类问题，否则不要修改业务代码。

## 背景说明

用户希望评估现有系统是否已经具备完整质量管理平台能力。当前平台包含测试执行、质量中心、AI 中枢、系统管理等模块，但需要确认：

- 各模块是否只是页面存在，还是能真实串联工作。
- 场景执行是否可以从用例/场景配置进入执行，并沉淀执行记录。
- 报告、缺陷、质量门禁是否能承接执行结果形成闭环。
- AI 中枢能力是否能支持用例、断言、失败分析、报告总结等质量工作。
- 距离“完整质量管理系统”还缺失哪些核心能力。

## 禁止事项

- 不做无关重构。
- 不直接重写页面样式、业务逻辑或 API。
- 不改变公开 API 响应结构。
- 不引入新依赖。

## 审计范围

### 1. 基础可运行性

- 前端是否能安装依赖、构建、启动。
- 后端是否能安装依赖、启动、通过健康检查。
- 前后端 API 代理、认证 token、环境配置是否能协同工作。
- 数据库初始化或迁移是否有清晰路径。

### 2. 用户与系统管理

验证以下模块是否具备基本可用能力：

- 登录/登出。
- 用户管理。
- 角色管理。
- 组织管理。
- 菜单管理。
- 字典管理。
- 日志管理。
- 权限控制是否真实影响菜单、按钮或接口访问。

### 3. 测试执行与用例管理

验证以下能力：

- 用例目录/文件夹管理。
- 接口用例创建、编辑、删除、查询。
- 请求方法、URL、Header、Body、断言、变量等配置是否完整。
- 用例单独执行是否可用。
- 执行结果是否展示状态、响应、耗时、错误原因。
- 功能用例或手工用例能力是否存在。
- 用例复制、导入导出、批量操作、标签、优先级等能力是否存在。

### 4. 场景管理与场景执行

这是本次最重要的链路，请尽量完成真实串联：

- 创建一个最小可执行用例。
- 创建一个场景。
- 将用例加入场景步骤。
- 配置步骤顺序、变量传递、失败策略或执行配置。
- 执行场景。
- 查看执行历史。
- 查看执行详情。
- 确认每个步骤的状态、耗时、请求响应、断言结果是否可追溯。
- 验证失败场景是否能记录失败原因。

如果无法执行，请明确说明阻塞在哪一层：

- 前端入口缺失。
- API 缺失。
- 参数结构不一致。
- 后端执行器异常。
- 环境/数据库/依赖问题。
- 缺少可执行测试目标。

### 5. 报告、缺陷与质量门禁闭环

验证以下能力是否完整：

- 执行结果是否能生成测试报告。
- 报告列表、详情、统计数据是否真实来自执行数据。
- 报告是否能关联场景、用例、执行记录。
- 缺陷是否能从失败结果创建或关联。
- 缺陷中心是否支持查询、新增、编辑、状态流转、严重程度、优先级、处理人、关闭。
- 质量门禁是否能基于执行结果、缺陷数量、通过率等规则判断通过/失败。
- 是否存在质量趋势、风险概览、版本质量评估能力。

### 6. AI 中枢

验证以下模块是否具备真实能力、是否能与质量链路结合：

- AI 模型配置。
- Prompt 模板管理。
- 变体生成。
- 断言生成。
- 失败分析。
- 报告总结。
- 建议历史。

重点判断：

- 是否能读取现有用例/执行/报告上下文。
- 是否能生成可落地的用例变体、断言、缺陷分析或报告总结。
- 生成结果是否能回写到业务模块。
- Prompt 模板是否可复用、可管理、可追踪。

### 7. 产品完整性分析

请从“完整质量管理平台”的角度分析当前缺口，至少覆盖：

- 需求管理/测试计划是否缺失。
- 测试版本、迭代、里程碑是否缺失。
- 测试套件、批量执行、定时执行、CI/CD 集成是否缺失。
- 测试环境管理、全局变量、密钥管理是否完整。
- 数据构造、前置/后置步骤、依赖步骤是否完整。
- 缺陷生命周期、责任人、评论、附件、变更历史是否完整。
- 报告维度、趋势分析、质量风险、覆盖率是否完整。
- 权限、审计、操作日志是否完整。
- 通知、Webhook、飞书/钉钉/邮件集成是否缺失。
- 可观测性、失败排查、日志下载是否完整。
- AI 能力是否只是工具页，还是已进入质量闭环。

## 验收标准

- 输出一份功能覆盖矩阵，包含模块、功能点、验证方式、结果、证据、问题编号。
- 至少完成一次场景执行链路验证；如无法完成，必须给出明确阻塞原因和最小复现步骤。
- 输出缺陷/阻塞问题清单，按 P0/P1/P2 排序。
- 输出“完整质量管理”缺失能力清单，按业务价值和实现优先级排序。
- 输出下一步建议，说明哪些内容适合拆成 Claude 实现任务包。
- 所有结论必须基于实际文件、接口、页面或测试结果，不确定的地方标记为“待确认”。

## 验证方式

优先运行：

```text
cd frontend && npm run build
cd backend && pytest
```

如果项目已有更细的测试命令，也请补充运行。建议尽量启动前后端并进行浏览器手工串联验证，尤其是场景执行链路。

如果命令无法运行，说明：

- 执行的命令。
- 失败原因。
- 是否是依赖、环境、数据库、配置、代码缺陷导致。
- 对审计结论的影响。
