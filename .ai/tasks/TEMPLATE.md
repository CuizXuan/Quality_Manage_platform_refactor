# Codex 实现任务包

## 任务目标

描述 Claude 需要实现的功能、修复的缺陷或执行的重构。

## 背景说明

说明用户侧问题、业务背景和相关产品行为。

## 视觉/交互基准页

- 如涉及前端页面、菜单、弹窗、抽屉、表格、查询栏、主题适配，必须写清楚参考页。
- 至少写一个明确基准，例如：
  - `frontend/src/views/scenario/ScenarioList.vue`
  - `frontend/src/views/case/CaseManagement.vue`
  - `frontend/src/app/AppShell.vue`
- 如果用户明确要求“和某页一致”，这里必须原样写出该页路径。

## 必读文件

- `AGENTS.md`
- 在这里列出相关后端文件。
- 在这里列出相关前端文件。
- 在这里列出相关测试文件。

## 允许修改范围

- 列出 Claude 可以修改的文件、目录或模块。

## 禁止事项

- 不做无关重构。
- 除非明确要求，不改变公开 API 响应结构。
- 除非明确要求，不引入新依赖。
- 未先说明原因前，不修改允许范围之外的文件。

## 实现要求

- 函数保持短小，职责单一。
- 路由函数保持轻薄。
- 业务逻辑放在 service 层。
- 前端 API 调用放在 `frontend/src/api/`。
- Vue 组件保持职责聚焦。
- 对用户输入进行校验。
- 保持任务范围外的现有行为不变。
- 对用户可见的说明、计划、总结、测试结果必须使用中文。
- 如涉及新增菜单或新增页面，先复制最接近的标准页结构/样式模式，再做定向改造。
- 不允许自行发明新的页面骨架、菜单组织方式、header/filter/table 视觉体系。
- 如涉及主题适配，必须同时检查深色和浅色分支，尤其 `html:not(.dark)`。
- 如涉及样式统一，必须明确指出“哪些实现直接参考基准页”。

## 验收标准

- 在这里写清楚可判断通过或失败的标准。
- 覆盖成功路径。
- 覆盖异常路径。
- 如果涉及前端，写清楚前端交互和展示行为。

## 验证方式

运行最相关的命令，例如：

```text
cd backend && pytest
cd frontend && npm run build
```

如果命令无法运行，说明阻塞原因和风险。

## Claude 输出要求

结束时必须用中文汇报：
- 变更文件。
- 已运行的测试或检查。
- 每项检查的结果。
- 剩余风险。
